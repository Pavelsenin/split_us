# Техническое задание

## 1. Назначение документа

Настоящий документ определяет технические требования к MVP-системе, позволяющей группе пользователей совместно вести общий чек и рассчитывать взаиморасчеты со строго оптимальным минимальным количеством переводов.

Система должна быть реализована как многокомпонентное решение, состоящее из:

- серверной части на Java;
- Telegram-бота как основного пользовательского клиента;
- отдельной web-админ-панели;
- дашбордов мониторинга на базе Grafana.

Документ предназначен для использования в качестве базового источника требований для проектирования, разработки, тестирования, развертывания и приемки MVP.

## 2. Цель продукта

Продукт должен позволять группе друзей, знакомых или коллег:

- создавать общий чек в Telegram;
- добавлять в чек зарегистрированных и незарегистрированных участников;
- фиксировать расходы, оплаченные разными участниками;
- делить каждый расход поровну между выбранными участниками;
- по запросу пересчитывать балансы после любых изменений;
- формировать строго оптимальный план взаиморасчетов с минимальным количеством переводов.

## 3. Объем MVP

### 3.1 Что входит в MVP

- Создание общего чека через Telegram-бота.
- Автоматическое создание отдельного Telegram-чата для каждого чека там, где это технически возможно.
- Резервный сценарий с `deep link` и инструкцией пользователю, если автоматическое создание чата технически невозможно.
- Управление участниками: зарегистрированными Telegram-пользователями и незарегистрированными участниками, существующими только в рамках чека.
- Публичная ссылка в закрепленном сообщении для добавления зарегистрированных пользователей.
- Ввод расходов свободным текстом и через меню/форму бота.
- Редактирование и удаление расходов только их автором.
- Пересчет балансов по явному запросу пользователя.
- Формирование строго оптимального расчета с использованием только точного алгоритма.
- Постоянное хранение чатов, участников, расходов, истории объединений и эксплуатационно значимых данных.
- Админ-панель для просмотра данных и удаления чеков.
- REST API для внутренней работы админ-панели и внутреннего повторного использования backend-возможностей.
- Логирование, мониторинг и резервное копирование.
- API-документация в формате OpenAPI/Swagger.
- Пользовательская документация на русском языке.
- Автоматическое тестирование, включая unit- и end-to-end тесты.

### 3.2 Что явно не входит в MVP

- Шифрование данных как отдельная функциональность.
- Несколько валют и конвертация валют.
- Сложные организационные группы и корпоративные модели доступа.
- Отслеживание погашения долгов.
- Категории расходов.
- Статусы жизненного цикла чека, такие как `draft`, `closed`, `archived`.
- Отмена объединения участников.
- Push, email, SMS и другие каналы уведомлений вне Telegram.
- Офлайн-режим сверх отложенной доставки сообщений клиентом Telegram.
- Добавление расходов задним числом.
- Публичный внешний API для сторонних интеграций.
- CI/CD.

## 4. Целевая аудитория

Основная аудитория:

- друзья, делящие общие расходы;
- знакомые, ведущие неформальные совместные траты;
- коллеги, делящие небольшие общие расходы без сложных корпоративных процессов.

Продукт предназначен для легковесного совместного использования в частных и неформальных группах, а не для публичных сообществ или сложного корпоративного учета.

## 5. Общий контекст системы

### 5.1 Основной интерфейс

Единственным пользовательским интерфейсом в MVP должен быть Telegram.

### 5.2 Компоненты системы

Система должна включать как минимум следующие компоненты:

- backend-приложение на Java, содержащее бизнес-логику и REST API;
- Telegram-бот, интегрированный с Telegram Bot API в режиме webhook;
- реляционную или эквивалентную постоянную базу данных для хранения бизнес-данных;
- web-админ-панель с отдельной аутентификацией;
- стек мониторинга с дашбордами Grafana;
- механизм резервного копирования базы данных.

### 5.3 Целевая среда развертывания

Решение должно разворачиваться на VDS.

Если для режима webhook это требуется, окружение должно предоставлять публичный HTTPS-домен, доступный для Telegram.

Telegram-бот должен иметь права администратора в групповом чате чека. Это обязательное эксплуатационное требование MVP.

## 6. Бизнес-сущности

MVP должно поддерживать как минимум следующие доменные сущности.

### 6.1 Пользователь

Зарегистрированный пользователь, идентифицируемый через Telegram и участвующий в одном или нескольких чеках.

Основные атрибуты:

- внутренний идентификатор;
- идентификатор пользователя Telegram;
- username в Telegram;
- временные метки регистрации;
- служебные метаданные.

Ограничение:

- пользователи без Telegram `username` не могут участвовать в чеке до тех пор, пока не настроят `username` в Telegram.

### 6.2 Чек

Совместное пространство учета расходов, привязанное к отдельному Telegram-чату.

Основные атрибуты:

- внутренний идентификатор;
- название чека;
- идентификатор владельца;
- идентификатор Telegram-чата;
- базовая валюта;
- временная метка создания;
- флаг активности или эквивалентный признак, что чат жив.

Примечание:

- данные о владельце должны синхронизироваться с Telegram там, где это возможно, но в MVP используются только для отладки и операционного просмотра.

### 6.3 Участник

Представляет человека, включенного в чек. Участник может быть:

- зарегистрированным, связанным с Telegram-пользователем;
- незарегистрированным, существующим только в рамках конкретного чека.

Основные атрибуты:

- внутренний идентификатор;
- идентификатор чека;
- отображаемое имя;
- тип участника;
- идентификатор связанного зарегистрированного пользователя, если применимо;
- метаданные происхождения объединения, если применимо;
- временная метка создания.

Ограничения:

- имена участников должны быть уникальны в пределах одного чека;
- для зарегистрированных пользователей каноническим именем участника считается Telegram `username`.

### 6.4 Расход

Представляет одну зафиксированную оплату.

Основные атрибуты:

- внутренний идентификатор;
- идентификатор чека;
- сумма;
- валюта;
- идентификатор участника-плательщика;
- краткий комментарий;
- исходный текст, если расход был создан из свободного текста;
- статус валидности;
- кто создал;
- кто обновил;
- временная метка создания;
- временная метка обновления.

### 6.5 Доля расхода

Представляет участника, включенного в конкретное распределение расхода.

Основные атрибуты:

- идентификатор расхода;
- идентификатор участника;
- рассчитанная доля суммы.

### 6.6 Запись об объединении участников

Представляет необратимое объединение незарегистрированного участника с зарегистрированным.

Основные атрибуты:

- идентификатор исходного участника;
- идентификатор целевого участника;
- идентификатор чека;
- кто выполнил объединение;
- временная метка.

## 7. Роли пользователей

### 7.1 Владелец чека

Зарегистрированный пользователь, который создает чек и становится владельцем соответствующего Telegram-чата.

Права:

- создает чек;
- автоматически добавляется как участник;
- может запрашивать расчет;
- имеет все возможности обычного зарегистрированного участника.

### 7.2 Зарегистрированный участник

Пользователь Telegram, добавленный в чат чека напрямую или по публичной ссылке.

Права:

- просматривать состояние чека;
- добавлять зарегистрированных и незарегистрированных участников в рамках бизнес-правил;
- добавлять расходы;
- редактировать только собственные расходы;
- удалять только собственные расходы;
- инициировать пересчет;
- объединять зарегистрированных и незарегистрированных участников.

### 7.3 Администратор

Внутренний операционный пользователь с доступом к админ-панели.

Права:

- проходить аутентификацию по логину и паролю;
- просматривать системные данные в режиме `read-only`;
- полностью удалять чеки;
- инициировать закрытие или деактивацию чата на уровне системы;
- просматривать технические и бизнес-метрики.

## 8. Функциональные требования

### 8.1 Создание чека

1. Система должна позволять зарегистрированному Telegram-пользователю запросить создание чека через бота.
2. Бот должен требовать от пользователя указать название чека при создании.
3. Система должна создавать отдельный Telegram-чат для чека там, где это технически возможно.
4. Если прямое автоматическое создание чата невозможно из-за ограничений платформы Telegram, система должна использовать резервный сценарий, в котором бот предоставляет `deep link` и инструкцию по ручному созданию группы и добавлению бота.
5. Создатель становится владельцем чека.
6. Создатель автоматически добавляется в чек как зарегистрированный участник.
7. Бот должен публиковать и закреплять сообщение в созданном чате.
8. Закрепленное сообщение должно содержать как минимум:
   - название чека;
   - краткую инструкцию по использованию;
   - публичную ссылку для добавления зарегистрированных пользователей.
9. Система должна ограничивать создание чеков одним пользователем не более чем 100 чеками за скользящее окно в 24 часа.

### 8.2 Управление участниками

1. Зарегистрированные участники должны иметь возможность присоединяться к чеку по публичной ссылке.
2. При присоединении по ссылке система должна автоматически, без дополнительных подтверждений, добавлять Telegram-пользователя в чек как зарегистрированного участника.
3. Для участия зарегистрированного пользователя обязателен Telegram `username`; пользователи без `username` должны получать ошибку и инструкцию настроить его перед присоединением.
4. Любой зарегистрированный участник чата чека должен иметь возможность добавить незарегистрированного участника, явно указав уникальное отображаемое имя.
5. Система должна отклонять создание участника, если запрошенное имя уже используется любым участником того же чека.
6. Если зарегистрированный пользователь не может быть добавлен из-за конфликта имени в чеке, система должна отклонять операцию с явной ошибкой.
7. Система должна поддерживать до 50 участников на один чек, включая зарегистрированных и незарегистрированных.
8. Любой зарегистрированный участник должен иметь возможность объединить незарегистрированного участника с зарегистрированным.
9. После объединения все исторические и текущие ссылки на незарегистрированного участника должны быть переназначены зарегистрированному.
10. После объединения результирующая сущность участника должна использовать представление и имя зарегистрированного пользователя.
11. Система должна сохранять историю объединений.
12. Операция объединения является необратимой в MVP.
13. Если Telegram `username` изменился и интеграция платформы передала обновленные данные, система должна по возможности автоматически обновлять имя участника в активных чеках.
14. Любой зарегистрированный участник должен иметь возможность переименовать незарегистрированного участника через бота.
15. После переименования имя незарегистрированного участника должно оставаться уникальным в пределах чека.
16. Участники не удаляются из чека в MVP и могут оставаться в нем с нулевым балансом.

### 8.3 Создание расходов

1. Любой зарегистрированный участник должен иметь возможность добавить расход.
2. Система должна поддерживать два режима создания расхода:
   - разбор свободного текстового сообщения;
   - структурированный сценарий через меню/форму.
3. Каждый расход должен включать:
   - сумму;
   - плательщика;
   - одного или более участников, на которых делится расход;
   - краткий комментарий.
4. Сумма должна поддерживать дробные значения.
5. Точность суммы должна быть ограничена двумя знаками после запятой.
6. Валюта должна быть фиксирована на уровне чека и в MVP равна российскому рублю.
7. У одного расхода должен быть ровно один плательщик.
8. Плательщик может входить или не входить в список участников, на которых делится расход.
9. Распределение расхода между выбранными участниками в MVP всегда выполняется поровну.
10. Система должна сохранять исходный свободный текст, если расход был создан из сообщения.
11. Расходы, созданные через форму, должны храниться в базе данных как источник истины и быть связаны с опубликованным ботом сообщением в Telegram.
12. Система должна поддерживать до 1000 расходов на один чек.

### 8.4 Поведение при разборе свободного текста

1. Бот должен принимать текстовые сообщения, предназначенные для описания расхода.
2. Бот должен пытаться извлечь:
   - сумму;
   - плательщика;
   - участников, на которых делится расход;
   - краткий комментарий, если его можно определить.
3. Если уверенности в разборе недостаточно или обязательные поля определить невозможно, бот не должен молча создавать расход.
4. В неоднозначных случаях бот должен попросить пользователя подтвердить данные или переформулировать сообщение.
5. Система должна сохранять исходный текст для отладки и анализа.
6. Если пользователь редактирует исходное Telegram-сообщение, связанное с расходом из свободного текста, система должна повторно валидировать или повторно разбирать это сообщение.
7. Если после редактирования сообщение становится невалидным или его нельзя однозначно разобрать, соответствующий расход должен быть переведен в состояние `invalid / requires clarification` и исключен из расчета до исправления.
8. Бот должен задать автору уточняющий вопрос и привести пример корректного сообщения.
9. Если исходное Telegram-сообщение, связанное с расходом из свободного текста, удалено, соответствующий расход должен быть автоматически удален.

### 8.5 Поведение сообщений для расходов, созданных через форму

1. Для расходов, созданных через форму бота, бот должен публиковать связанное Telegram-сообщение, представляющее созданный расход.
2. Источником истины остается запись расхода в базе данных.
3. Редактирование расхода, созданного через форму, должно выполняться через бота, после чего бот должен обновлять связанное сообщение.
4. Если связанное сообщение расхода, созданного через форму, редактируется, система должна повторно разобрать или повторно провалидировать его.
5. Если после редактирования связанное сообщение расхода, созданного через форму, становится невалидным или его нельзя однозначно разобрать, соответствующий расход должен быть переведен в состояние `invalid / requires clarification` и исключен из расчета до исправления.
6. Если связанное сообщение расхода, созданного через форму, удалено, соответствующий расход должен быть автоматически удален.
7. Основным сценарием удаления расхода, созданного через форму, должно быть удаление связанного сообщения бота автором расхода.
8. Если ограничения Telegram не позволяют автору напрямую удалить связанное сообщение бота, система должна предоставлять команду или кнопку бота, которая удаляет расход и затем удаляет либо помечает связанное сообщение соответствующим образом.

### 8.6 Редактирование и удаление расходов

1. Любой зарегистрированный участник чека может редактировать только расходы, созданные им самим.
2. Любой зарегистрированный участник чека может удалять только расходы, созданные им самим.
3. Допустимые для редактирования поля:
   - сумма;
   - плательщик;
   - участники, на которых делится расход;
   - комментарий.
4. Все изменения должны сохраняться вместе с метаданными обновления.
5. Система должна гарантировать, что в расчете участвует только последняя валидная версия расхода.
6. Исправлять невалидные расходы может только их автор.

### 8.7 Пересчет и текущее состояние балансов

1. Система должна позволять выполнять расчет в любой момент, пока чат чека активен.
2. Система не должна автоматически публиковать результаты расчета после каждого нового или измененного расхода.
3. Результаты расчета публикуются только по явному запросу пользователя через команду или действие в меню бота.
4. Вывод расчета должен включать:
   - сколько заплатил каждый участник;
   - сколько должен каждый участник;
   - итоговый баланс каждого участника;
   - список переводов для взаиморасчетов.
5. Система не должна хранить отдельную историю снимков расчетов в MVP.
6. Для одного чека одновременно может выполняться только один расчет.
7. Если расчет уже выполняется, повторные запросы расчета должны отклоняться сообщением о том, что расчет уже идет.
8. Если во время выполнения расчета данные чека изменились, текущий расчет должен быть прерван, а пользователи должны заново запросить новый расчет.
9. Если расчет выполняется дольше ожидаемого интерактивного времени, бот должен ответить, что расчет выполняется, а итоговый результат опубликовать позже.

### 8.8 Алгоритм расчета

1. Система должна рассчитывать взаиморасчеты на основе всех текущих валидных расходов в чеке.
2. Результат должен строго минимизировать количество переводов, а не приближенно.
3. Система должна формировать корректный план переводов в формате `должник -> получатель`.
4. Набор переводов должен полностью удовлетворять балансам участников с учетом правил округления.
5. Решение должно корректно работать с дробными значениями, округленными до двух знаков после запятой.
6. Алгоритм должен быть детерминированным при одинаковых входных данных.
7. Реализация должна использовать только точный алгоритм и не должна использовать приближенные или запасные стратегии оптимизации.
8. Система должна публиковать длительность расчета как метрику.
9. Должен быть настроен алерт, если длительность расчета превышает 1 секунду.
10. Если при равном делении суммы на двухзнаковой точности возникает остаток, этот остаток должен детерминированно назначаться первому участнику из списка получателей по внутреннему порядку участников в чеке.

### 8.9 Итоговый расчет и удаление чека

1. Чек остается активным, пока активен связанный Telegram-чат, если только чек не был удален администратором.
2. В MVP чек не использует явные статусы жизненного цикла, такие как `draft`, `closed`, `archived`.
3. Пользовательская операция получения итоговых взаиморасчетов представляет собой только запрос расчета; формирование расчета не закрывает чат и не завершает чек.
4. Администратор должен иметь возможность полностью удалить чек через админ-панель.
5. Административное удаление должно включать закрытие или деактивацию связанного Telegram-чата через интеграцию с ботом/системой там, где это поддерживается Telegram и правами доступа.
6. Если удалить или закрыть чат на стороне Telegram невозможно из-за ограничений API, допустимым резервным поведением считается логическое закрытие чека в системе без физического удаления чата.

### 8.10 Ограничения и правила валидации

Система должна обеспечивать как минимум следующие ограничения:

- не более 100 созданных чеков одним пользователем за 24-часовое окно;
- не более 50 участников в одном чеке;
- не более 1000 расходов в одном чеке;
- уникальность имен участников в пределах одного чека;
- ровно один плательщик на один расход;
- хотя бы один участник в списке распределения расхода;
- только положительная сумма расхода;
- не более двух знаков после запятой.

## 9. Требования к Telegram-боту

### 9.1 Общие требования

1. Бот должен быть основным пользовательским интерфейсом.
2. Бот должен работать через webhook.
3. Окружение развертывания должно поддерживать настройку HTTPS webhook endpoint.
4. Интерфейс бота в MVP должен быть на русском языке.

### 9.2 Поддерживаемые сценарии бота

Бот должен поддерживать как минимум:

- сценарий создания чека;
- инструкцию по приглашению участников;
- добавление незарегистрированных участников;
- запуск объединения участников;
- добавление расходов свободным текстом;
- добавление расходов через структурированный сценарий;
- редактирование собственных расходов;
- удаление собственных расходов;
- запрос текущего баланса;
- запрос расчета.

### 9.3 Сообщения в чате

Бот должен публиковать в чате чека сообщения в формате, достаточном для валидации участниками:

- текущего состава участников;
- зафиксированных расходов;
- текущих балансов;
- результата расчета;
- сервисных уведомлений о создании, обновлении, удалении, инвалидировании расходов и об автоматическом удалении расхода вследствие удаления связанного сообщения.

## 10. Требования к backend

### 10.1 Технологическое ограничение

Backend должен быть реализован на Java.

### 10.2 API

1. Backend должен предоставлять REST API, необходимый для:
   - функций админ-панели;
   - внутренней операционной интеграции;
   - возможного будущего внутреннего повторного использования управления чеками и расчетами.
2. API должен быть задокументирован в формате OpenAPI/Swagger.
3. API должен возвращать машиночитаемые ответы об ошибках.
4. В MVP REST API предназначен только для внутреннего использования.
5. Для внутренней аутентификации API должен использоваться отдельный `service token`, а не токен Telegram-бота.

### 10.3 Хранение данных

1. Система должна сохранять все критичные бизнес-данные в постоянной базе данных.
2. Выбранное хранилище должно поддерживать:
   - транзакционную целостность при обновлении чеков и расходов;
   - выборки участников и расходов по чеку;
   - сохранение истории объединений;
   - административный поиск и удаление.

## 11. Требования к админ-панели

### 11.1 Общие требования

1. Админ-панель должна быть отдельным web-интерфейсом.
2. Админ-панель должна требовать аутентификацию по логину и паролю.

### 11.2 Функциональный объем

Админ-панель должна предоставлять как минимум:

- вход администратора;
- поиск и просмотр чеков;
- `read-only` доступ к названию чека;
- `read-only` доступ к участникам;
- `read-only` доступ к связям объединений;
- `read-only` доступ к расходам;
- `read-only` доступ к информации о владельце чека;
- `read-only` доступ к исходным текстам сообщений, на основе которых были созданы расходы;
- возможность полностью удалить чек;
- возможность инициировать операционное закрытие или деактивацию связанного Telegram-чата там, где это технически возможно;
- доступ к связанным метрикам или встроенным мониторинговым представлениям.

### 11.3 Ограничения административных действий

Следующие действия не требуются в MVP, если только не понадобятся как внутренняя реализационная деталь:

- ручное редактирование расходов;
- ручное редактирование состава участников;
- ручное изменение объединений участников.

Единственным обязательным бизнес-действием с изменением данных в админ-панели является полное удаление чека с сопутствующей очисткой.

## 12. Мониторинг и логирование

### 12.1 Стек мониторинга

Для визуализации дашбордов должна использоваться Grafana.

### 12.2 Обязательные технические метрики

Система должна публиковать или собирать как минимум следующие технические метрики:

- загрузка CPU;
- использование памяти;
- ошибки обработки запросов;
- скорость поступления запросов;
- метрики времени ответа;
- индикаторы доступности и health-check.

### 12.3 Обязательные бизнес-метрики

Система должна публиковать или собирать как минимум следующие бизнес-метрики:

- количество активных пользователей;
- количество входящих запросов;
- количество созданных чеков;
- количество активных чеков;
- количество добавленных расходов;
- количество запросов на расчет;
- длительность расчета.

### 12.4 Логирование

Система должна формировать и хранить:

- системные логи;
- бизнес-логи;
- логи ошибок интеграции с Telegram.

Рекомендуемый состав бизнес-событий для логирования:

- расчет сформирован;
- чек удален.

## 13. Требования к безопасности

Требования безопасности в MVP намеренно минимальны, но система должна обеспечивать базовые меры:

- аутентифицированную идентификацию Telegram-пользователей по данным платформы Telegram;
- аутентификацию в админ-панели по логину и паролю;
- проверки авторизации, гарантирующие, что только участники чека могут изменять его данные;
- проверки авторизации, гарантирующие, что только автор расхода может редактировать или удалять свой расход;
- проверки авторизации, гарантирующие, что только администраторы имеют доступ к операциям админ-панели;
- защищенный транспорт по HTTPS для webhook и админ-панели;
- безопасное хранение паролей администраторов с использованием современного алгоритма хеширования;
- аутентификацию внутреннего REST API через `service token`.

Отдельная функциональность шифрования данных не требуется, помимо стандартной защиты транспорта и учетных данных.

## 14. Работа с данными и приватность

1. Система должна хранить пользовательский контент, который потенциально может содержать персональные или чувствительные данные.
2. Система должна по возможности избегать избыточного дублирования данных.
3. Система должна сохранять исходный текст сообщения, если он использовался для создания расхода, для целей отладки.
4. Система должна поддерживать резервное копирование базы данных.
5. Система должна поддерживать административное удаление данных на уровне чека.
6. Срок хранения логов должен составлять 7 дней.
7. Для записей чеков и расходов в MVP не устанавливается автоматический срок удаления.

## 15. Нефункциональные требования

### 15.1 Надежность

- Система должна сохранять целостность данных при операциях над участниками и расходами.
- Результаты расчета должны быть воспроизводимыми при одинаковых входных данных.
- Ошибки доставки сообщений в Telegram должны логироваться и быть доступны для операционного анализа.

### 15.2 Производительность

Формальные требования по пропускной способности в MVP не зафиксированы, однако реализация должна оставаться отзывчивой при следующих ограничениях:

- до 50 участников в одном чеке;
- до 1000 расходов в одном чеке.

Строго оптимальный расчет должен завершаться за приемлемое интерактивное время в рамках указанных ограничений.

Должен срабатывать алерт, если длительность расчета превышает 1 секунду.

### 15.3 Локализация

- Пользовательский интерфейс Telegram-бота должен быть на русском языке.
- Пользовательская документация должна быть на русском языке.

### 15.4 Эксплуатация

- Система должна развертываться на VDS.
- Система должна поддерживать мониторинг, сбор логов и резервное копирование.
- Резервное копирование базы данных должно выполняться один раз в сутки.
- Резервные копии должны храниться 7 дней.
- Должна быть выполнена однократная проверка восстановления из резервной копии перед вводом MVP в эксплуатацию.
- Максимально допустимое время восстановления должно составлять 1 час.
- Восстановление из резервной копии выполняет администратор.

## 18. Критерии приемки

### 18.1 Создание чека

- Telegram-пользователь может создать чек, указав название.
- Создается отдельный Telegram-чат, либо предоставляется fallback-сценарий через `deep link`, если автоматическое создание невозможно.
- Создатель добавляется как зарегистрированный участник.
- В чате создается закрепленное сообщение со ссылкой приглашения.
- Система блокирует создание новых чеков после превышения суточного лимита.

### 18.2 Присоединение зарегистрированного участника

- Зарегистрированный Telegram-пользователь с `username` может присоединиться по ссылке.
- Пользователь автоматически становится участником правильного чека.
- Повторная регистрация одного и того же участника предотвращается.
- Пользователь без Telegram `username` получает понятную ошибку.

### 18.3 Добавление незарегистрированного участника

- Любой зарегистрированный участник может добавить незарегистрированного участника.
- Добавление проходит успешно только при уникальном имени внутри чека.
- При превышении лимита участников добавление блокируется.

### 18.4 Объединение участников

- Зарегистрированный участник может объединить незарегистрированного участника с зарегистрированным.
- Все существующие ссылки в расходах корректно переназначаются.
- История объединений сохраняется.
- Объединение нельзя отменить через интерфейс MVP.

### 18.5 Создание расхода из свободного текста

- Участник может отправить корректно разбираемое текстовое сообщение о расходе.
- Система создает расход с корректными плательщиком, участниками, суммой и комментарием там, где это применимо.
- Исходный текст сообщения сохраняется.
- Если сообщение неоднозначно, бот запрашивает уточнение вместо создания некорректных данных.

### 18.6 Создание расхода через структурированный сценарий

- Участник может создать расход через форму или меню бота.
- Валидация не позволяет создать расход с некорректной суммой, пустым списком получателей или неизвестными участниками.
- Поддерживаются дробные суммы с точностью до двух знаков после запятой.

### 18.7 Редактирование и удаление расходов

- Любой зарегистрированный участник может редактировать только собственный расход.
- Любой зарегистрированный участник может удалять только собственный расход.
- Последующий расчет учитывает только последнюю валидную версию расхода.

### 18.8 Выполнение расчета

- Участник может запросить расчет в любой момент.
- Ответ содержит, сколько заплатил каждый участник, сколько должен, итоговый баланс и список переводов.
- Расчет можно повторять после любого изменения данных.
- Результат расчета публикуется только по явному запросу.
- Пока один расчет выполняется, второй не запускается и получает сообщение о том, что расчет уже выполняется.

### 18.9 Оптимальность расчета

- Для поддерживаемых лимитов система формирует корректный план взаиморасчетов с минимально возможным числом переводов.
- Результаты детерминированы при одинаковых входных данных.
- Балансы остаются корректными после округления до двух знаков.
- Реализация использует только точную оптимизацию без приближенного fallback.

### 18.10 Обработка редактирования и удаления связанных сообщений

- Если пользователь редактирует исходное Telegram-сообщение расхода и после редактирования оно остается валидным, данные расхода обновляются.
- Если после редактирования сообщение становится невалидным или неоднозначным, расход переводится в `invalid / requires clarification` и исключается из расчета.
- Бот запрашивает у автора уточнение и приводит пример корректного сообщения.
- Если связанное сообщение расхода удаляется, соответствующий расход автоматически удаляется.

### 18.11 Админ-панель

- Администратор может пройти аутентификацию по логину и паролю.
- Администратор может просматривать данные чека в режиме `read-only` и удалять чек.
- Удаление чека запускает соответствующую очистку и операционное закрытие или деактивацию.
- Мониторинговые представления доступны администратору.

### 18.12 Мониторинг и логи

- Технические дашборды показывают состояние серверной части.
- Бизнес-дашборды показывают эксплуатационные показатели, включая длительность расчета.
- Системные, бизнес- и Telegram-интеграционные логи формируются и доступны для эксплуатации.

### 18.13 Резервное копирование

- Существует механизм резервного копирования базы данных, который можно успешно выполнить.
- Резервное копирование выполняется раз в сутки.
- Резервные копии хранятся 7 дней.
- Процедура резервного копирования и восстановления задокументирована.
- Перед вводом в эксплуатацию выполнена проверка восстановления из резервной копии.

## 19. Открытые проектные решения

Следующие пункты сознательно оставлены на этап проектирования, оставаясь в рамках данного ТЗ:

- конкретный механизм резервного копирования.

## 20. Рекомендуемые зоны покрытия API

Внутренний backend API должен как минимум покрывать:

- создание и получение чеков;
- управление участниками;
- операции объединения участников;
- CRUD для расходов;
- получение результата расчета;
- административное удаление чеков;
- health и интеграцию с метриками там, где применимо.

Финальный дизайн endpoint-ов определяется на этапе проектирования и документируется в OpenAPI/Swagger.

## 21. Сводка обязательных ограничений

- Пользовательский интерфейс: Telegram-бот.
- Язык backend: Java.
- Дашборды мониторинга: Grafana.
- Админ-панель: отдельный web-интерфейс.
- Среда развертывания: VDS.
- Транспорт бота: webhook по HTTPS.
- Язык бота: русский.
- Язык пользовательской документации: русский.
- Права администратора у бота в групповом чате: обязательны.
- Валюта MVP: только RUB.
- Распределение расхода: только поровну.
- Один расход: только один плательщик.
- Редактирование и удаление расходов: только своих.
- Уникальность имен участников в пределах чека: обязательна.
- Telegram `username` обязателен для зарегистрированного участия.
- Строго оптимальный расчет с минимальным числом переводов: обязателен.
- Только точный алгоритм без fallback: обязателен.
- REST API: только внутренний.
- Внутренняя аутентификация API: отдельный `service token`.
- Автотесты: unit и end-to-end обязательны.
- Документация API: OpenAPI/Swagger обязательна.
- Ежедневные резервные копии с хранением 7 дней: обязательны.
