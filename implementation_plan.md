# План реализации MVP проекта

## 1. Цель плана

Документ переводит требования из [requirements.md](/D:/Dev/projects/codex/split_us/requirements.md) в последовательный план реализации MVP.

План рассчитан на ситуацию, когда в репозитории еще нет готового прикладного кода, и сначала нужно спроектировать архитектуру, затем последовательно собрать backend, Telegram-бота, админ-панель и эксплуатационный контур.

## 2. Исходные ограничения из ТЗ

- основной пользовательский интерфейс: Telegram-бот;
- backend: Java;
- бот работает через webhook по HTTPS;
- админ-панель: отдельный web-интерфейс;
- мониторинг: Grafana;
- развертывание: VDS;
- валюта MVP: только RUB;
- распределение расхода: только поровну;
- один расход: один плательщик;
- REST API: только внутренний;
- обязательны unit- и end-to-end тесты;
- обязательны OpenAPI/Swagger, пользовательская документация, логирование, мониторинг и резервное копирование.

## 3. Главные проектные риски

### 3.1 Критический риск

ТЗ требует строго оптимальный расчет с минимальным числом переводов и без приближенного fallback. Это нужно считать главным техническим риском проекта.

До начала основной разработки нужен отдельный spike, который:

- фиксирует математическую постановку задачи;
- выбирает точный алгоритм;
- проверяет детерминированность;
- проверяет время работы на целевых ограничениях MVP;
- определяет допустимую стратегию отмены или прерывания расчета при изменении данных.

Если spike показывает, что требование не укладывается в интерактивное время на лимитах MVP, это нужно поднимать как проектное решение до начала основной реализации.

### 3.2 Остальные риски

- ограничения Telegram на создание и сопровождение групповых чатов потребуют fallback-сценария через `deep link`;
- обработка редактирования и удаления Telegram-сообщений влияет на консистентность расходов;
- нужны строгие правила авторизации на уровне чека и на уровне автора расхода;
- в ТЗ оставлены открытыми ключевые выборы: Java-фреймворк, СУБД, библиотека бота, frontend-стек, схема хранения, webhook security, схема VDS и backup-механизм.

## 4. Этапы реализации

## Этап 0. Проектирование и закрытие открытых решений

Цель: снять архитектурную неопределенность до начала основной разработки.

Задачи:

- выбрать Java-фреймворк для backend;
- выбрать СУБД;
- выбрать библиотеку или SDK для Telegram Bot API;
- выбрать frontend-стек админ-панели;
- спроектировать доменную модель и схему хранения;
- определить модель webhook security;
- определить схему развертывания на VDS;
- определить механизм ежедневных backup и восстановления;
- провести spike по точному алгоритму расчета.

Артефакты:

- 5-9 ADR или единый архитектурный документ;
- схема компонентов;
- ER-диаграмма или описание таблиц;
- draft OpenAPI;
- описание Telegram-сценариев;
- протокол spike по алгоритму расчета с benchmark.

Критерий завершения:

- все пункты из раздела 19 ТЗ закрыты конкретными решениями;
- есть подтверждение, что расчет можно реализовать в рамках ограничений MVP;
- утвержден критический путь разработки.

## Этап 1. Базовый каркас решения

Цель: подготовить рабочую основу для параллельной разработки.

Задачи:

- создать структуру репозитория или модулей;
- поднять Java backend с базовой конфигурацией;
- подключить БД и миграции;
- подготовить webhook endpoint для Telegram;
- подготовить заготовку админ-панели;
- добавить конфигурацию окружений через переменные среды;
- настроить базовые health-check endpoints;
- подготовить локальный docker-compose для разработки и тестов.

Артефакты:

- стартовый backend-проект;
- миграции схемы БД;
- каркас админ-панели;
- локальное dev-окружение;
- шаблон конфигурации.

Критерий завершения:

- backend стартует локально;
- БД поднимается и применяет миграции;
- Telegram webhook технически готов принимать обновления;
- админ-панель открывается и может аутентифицироваться против заглушки или backend.

## Этап 2. Домен, хранение и внутренний API

Цель: реализовать устойчивое ядро бизнес-данных и внутренние контракты.

Задачи:

- реализовать сущности `user`, `check`, `participant`, `expense`, `expense_share`, `participant_merge`;
- ввести ограничения на уникальность имени участника, лимиты и целостность ссылок;
- реализовать внутренний REST API для чеков, участников, расходов, объединений и расчета;
- реализовать отдельную аутентификацию API через `service token`;
- подготовить машиночитаемые ошибки;
- собрать OpenAPI-описание.

Артефакты:

- миграции и индексы;
- доменная модель;
- CRUD и service-слой;
- OpenAPI/Swagger.

Критерий завершения:

- API покрывает минимум, перечисленный в разделе 20 ТЗ;
- правила валидации вынесены в backend и покрыты unit-тестами;
- админ-панель и бот могут опираться на единый внутренний API.

## Этап 3. Telegram-бот: создание чека и участники

Цель: довести до рабочего состояния базовый жизненный цикл чека в Telegram.

Задачи:

- реализовать регистрацию пользователя по данным Telegram;
- реализовать создание чека через бота;
- реализовать автоматическое создание чата там, где это возможно;
- реализовать fallback через `deep link` и инструкцию;
- публиковать и закреплять стартовое сообщение;
- реализовать присоединение зарегистрированных участников по ссылке;
- реализовать проверку наличия Telegram `username`;
- реализовать добавление и переименование незарегистрированных участников;
- реализовать объединение незарегистрированного и зарегистрированного участника;
- вести историю объединений;
- реализовать лимиты: 100 чеков за 24 часа и 50 участников на чек.

Покрываемые критерии приемки:

- 18.1;
- 18.2;
- 18.3;
- 18.4.

Критерий завершения:

- пользователь может создать чек и собрать состав участников полностью через Telegram;
- в базе данных фиксируются все связи и лимиты;
- ошибки и ограничения объясняются пользователю на русском языке.

## Этап 4. Telegram-бот: расходы и синхронизация с сообщениями

Цель: реализовать полный контур добавления и сопровождения расходов.

Задачи:

- реализовать создание расхода через меню или форму;
- реализовать разбор свободного текстового сообщения;
- сохранять исходный текст сообщения;
- реализовать подтверждение или запрос уточнения при неоднозначном разборе;
- реализовать публикацию связанного сообщения бота;
- реализовать редактирование только своих расходов;
- реализовать удаление только своих расходов;
- реализовать повторную валидацию при редактировании исходного Telegram-сообщения;
- реализовать перевод расхода в `invalid / requires clarification`;
- исключать невалидные расходы из расчета;
- автоматически удалять расход при удалении связанного сообщения.

Покрываемые критерии приемки:

- 18.5;
- 18.6;
- 18.7;
- 18.10.

Критерий завершения:

- оба сценария создания расхода работают;
- Telegram-сообщения и backend-состояние не расходятся;
- авторские ограничения подтверждены тестами.

## Этап 5. Движок расчета и публикация результата

Цель: реализовать строго оптимальный и детерминированный расчет.

Задачи:

- реализовать сбор текущих валидных расходов по чеку;
- реализовать расчет фактически оплаченного, долей и итоговых балансов;
- реализовать точный алгоритм минимизации числа переводов;
- обеспечить корректное округление до двух знаков;
- детерминированно назначать остаток первому участнику по внутреннему порядку;
- реализовать защиту от параллельных расчетов по одному чеку;
- реализовать отмену или отклонение расчета при изменении данных во время выполнения;
- публиковать результат только по явному запросу;
- реализовать отложенную публикацию для длинного расчета;
- публиковать метрику длительности расчета.

Покрываемые критерии приемки:

- 18.8;
- 18.9.

Критерий завершения:

- для тестовых и граничных наборов данных результат корректен, детерминирован и минимален по числу переводов;
- при конкурирующих запросах не возникает гонок и двойного запуска.

## Этап 6. Админ-панель

Цель: дать администраторам отдельный read-only доступ к данным и операцию удаления чека.

Задачи:

- реализовать аутентификацию администратора по логину и паролю;
- реализовать поиск чеков;
- показать read-only данные чека, участников, объединений, расходов и владельца;
- показывать исходные тексты сообщений;
- реализовать полное удаление чека;
- запускать закрытие или деактивацию связанного Telegram-чата там, где это возможно;
- встроить ссылки или представления мониторинга.

Покрываемые критерии приемки:

- 18.11.

Критерий завершения:

- администратор может найти любой чек, просмотреть данные и безопасно удалить его;
- операция удаления оставляет систему в консистентном состоянии.

## Этап 7. Мониторинг, логирование, безопасность и backup

Цель: закрыть обязательные эксплуатационные требования MVP.

Задачи:

- собрать технические и бизнес-метрики из раздела 12;
- настроить Grafana dashboards;
- настроить alert на длительность расчета более 1 секунды;
- реализовать системные, бизнес- и Telegram-интеграционные логи;
- обеспечить HTTPS для webhook и админ-панели;
- хранить пароли администраторов в виде безопасных хешей;
- реализовать ежедневные backup БД;
- настроить хранение backup и логов 7 дней;
- документировать восстановление;
- провести одно контрольное восстановление из резервной копии.

Покрываемые критерии приемки:

- 18.12;
- 18.13.

Критерий завершения:

- есть наблюдаемость по сервису, Telegram-интеграции и бизнес-операциям;
- резервные копии создаются автоматически и восстановление проверено.

## Этап 8. Тестирование, документация и приемка

Цель: довести решение до состояния, пригодного к выпуску MVP.

Задачи:

- собрать полный набор unit-тестов для расчета, валидации, прав доступа, парсинга, объединения и инвалидирования;
- реализовать end-to-end тесты ключевых Telegram- и admin-сценариев;
- покрыть позитивные, негативные и граничные кейсы из раздела 16;
- подготовить пользовательскую документацию на русском языке;
- финализировать OpenAPI/Swagger;
- подготовить инструкцию по развертыванию;
- подготовить инструкцию по backup и восстановлению;
- провести приемку по разделу 18.

Критерий завершения:

- все критерии приемки из раздела 18 проверены;
- документация достаточна для запуска и сопровождения MVP;
- выпуск не зависит от незафиксированных архитектурных решений.

## 5. Рекомендуемый порядок работ

Критический путь:

1. Этап 0.
2. Этап 1.
3. Этап 2.
4. Этап 3.
5. Этап 4.
6. Этап 5.
7. Этап 8.

Параллельно после стабилизации API можно вести:

- Этап 6 после завершения основной части Этапа 2;
- Этап 7 частично после Этапа 1, полностью после Этапа 5;
- часть тестов из Этапа 8 начиная с Этапа 2, а не в самом конце.

Практически это означает:

- сначала нужно доказать реализуемость точного алгоритма;
- затем закрепить схему данных и внутренний API;
- только после этого безопасно распараллеливать бот, админку и эксплуатационный контур.

## 6. Рекомендуемые контрольные точки

### Milestone 1. Architecture Baseline

Готово, если:

- выбраны стек и схема хранения;
- есть benchmark по алгоритму;
- утверждены API-контракты и схема VDS.

### Milestone 2. Telegram Core

Готово, если:

- работает создание чека;
- участники добавляются и объединяются;
- соблюдаются лимиты и авторизация.

### Milestone 3. Expense Flow

Готово, если:

- работают оба сценария создания расходов;
- поддерживаются редактирование, удаление и инвалидирование;
- Telegram-сообщения синхронизированы с backend.

### Milestone 4. Settlement Engine

Готово, если:

- расчет детерминирован;
- число переводов минимально;
- расчет устойчив к параллельным запросам и изменениям данных.

### Milestone 5. Operations Ready

Готово, если:

- админ-панель работает;
- Grafana и alerting настроены;
- backup и восстановление проверены;
- документация завершена.

## 7. Оценка по срокам

Если команда небольшая, реалистично планировать MVP как 8 этапов на 8-12 недель с обязательным резервом на spike алгоритма и Telegram-интеграцию.

Ориентир без детальной декомпозиции по людям:

- Этап 0: 1-2 недели;
- Этап 1-2: 1-2 недели;
- Этап 3-4: 2-4 недели;
- Этап 5: 1-2 недели;
- Этап 6-7: 1-2 недели;
- Этап 8: 1-2 недели.

Если spike по точному алгоритму или ограничения Telegram дадут негативный результат, сроки нужно пересматривать сразу, а не в конце проекта.

## 8. Что имеет смысл сделать сразу после утверждения плана

1. Зафиксировать решения по стеку и инфраструктуре в ADR.
2. Выполнить spike по точному алгоритму с тестовыми наборами данных уровня MVP.
3. Подготовить доменную схему и draft OpenAPI.
4. Разбить этапы 1-8 на backlog задач с приемкой по разделу 18 ТЗ.
