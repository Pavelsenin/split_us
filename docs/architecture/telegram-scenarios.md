# Telegram Scenarios

## 1. Создание чека

1. Пользователь пишет боту `/new_check`.
2. Бот запрашивает название чека.
3. Backend создает `check_book` и participant-владельца.
4. Если Telegram позволяет автоматическое создание чата, бот создает чат и публикует закрепленное сообщение.
5. Если нет, бот возвращает `deep link` и инструкцию по ручной сборке группы.

## 2. Присоединение зарегистрированного участника

1. Пользователь открывает ссылку из закрепленного сообщения.
2. Backend валидирует check token и Telegram `username`.
3. Если имя уникально и лимит не превышен, создается registered participant.
4. При конфликте имени или отсутствии `username` пользователь получает объяснимую ошибку на русском языке.

## 3. Добавление гостя

1. Зарегистрированный участник вызывает команду добавления.
2. Бот запрашивает имя гостя.
3. Backend проверяет уникальность имени в пределах чека и лимит 50 участников.
4. Создается `GUEST` participant.

## 4. Добавление расхода

Есть два входа:

- свободный текст в чат;
- структурированная форма через callback/menu.

В обоих случаях backend формирует `expense` и набор `expense_share`. При неоднозначном разборе создается запрос на уточнение, а расход не считается валидным до исправления.

## 5. Редактирование или удаление Telegram-сообщения

1. Telegram присылает webhook update о редактировании или удалении.
2. Backend по `telegram_chat_id + telegram_message_id` находит связанный расход.
3. При успешном повторном разборе обновляет расход.
4. При неуспешном переводит расход в `REQUIRES_CLARIFICATION` или удаляет, если сообщение удалено.

## 6. Запрос расчета

1. Участник явно запрашивает расчет.
2. Backend агрегирует только `VALID` расходы в балансы участников.
3. Exact solver строит минимальный по числу переводов план.
4. Результат публикуется в чат только после завершения расчета.
